# グローバルレポートスイートの使用

**WHAT:** サイトごとにレポートスイートを作成するのは魅力的ですが、レポート作成や実装が複雑になるという点では、すぐに最悪の事態に陥る可能性があります。 単一のグローバルレポートスイートを構築することは、多くの点で役立ち、実装を簡素化します。

**理由：** グローバルレポートスイートを作成することは、各プロパティのデジタルプロパティとユーザージャーニーの統一されたビューを持つための唯一のオプションです。 モバイルアプリとweb サイトがある場合は、アプリのデータとweb データを1つのレポートスイートにまとめて、クロスデバイスジャーニーを活用し、両方のプロパティを単一の訪問者として訪問する人と別々のレポートスイートとして訪問する人を追跡する必要があります。別々のレポートスイートでは、2人の訪問者 – 各プロパティに1人 – を示す不統一なデータセットが表示されます。

単一のレポートスイートを導入するメリットとデメリットを次に示します。

* 利点：
   * デジタル全体の状況を簡単に把握する能力。 他のヒントで参照されている「プロパティ」ディメンション（eVar）を実装した場合、すべてのサイトとアプリ、トラフィック、コンバージョンの単一のビューを簡単に取得できます。 このような全体像を把握することは、ビジネス全体を把握する上で重要です。
   * 同じ意味で、あらゆるプロパティをまたいで顧客がどのように移動しているのかを把握し、デジタル環境全体におけるジャーニーを把握できるようになりました。
   * 管理のしやすさ： 複数のレポートスイートを使用する場合は、複数の場所でインターフェイスを管理し、複数のタグ付けドキュメント（またはより複雑なドキュメント）を管理する必要があります。 すべてを一元管理することで、更新する場所は一つだけです。 また、アクセスを許可するのもずっと簡単です。
   * インターフェイスの使いやすさ： オーディエンスが必要なレポートスイートをひとつしか持っていない場合は、どのレポートスイートを選択すべきかを考える必要はありません。 同じワークスペースパネル内で複数のレポートスイートを使用することはできず、複数のレポートスイートを使用するとユーザーが混乱する可能性があることに注意してください。
   * サーバーコールが少ない=コストが少ない。 複数のレポートスイートを呼び出す場合、コストが増加します。 導入をシンプルにすることで、コストも削減できます。
   * 仮想レポートスイート（VRS）を活用するだけで、グローバルレポートスイート内のサイト固有のデータを分割し、必要に応じてVRSに基づいてユーザー権限を絞り込むことができます。 データを個々のレポートスイートに分割すると、ロールアップできませんが、既に1つのデータセット（グローバル RS）に結合されている場合は、簡単に分割できます。
* 短所：
   * ユーザーが一方から他方に切り離さない非常に個別のプロパティがあり、そうすることが期待されない場合は、別のレポートスイートを維持することをお勧めします。
   * プロパティのタグ付けとレポートのニーズが大きく異なる場合は、効率を調整するために、レポートスイートを個別に設定するのが理にかなっています。 別のレポートスイートを使用すると、カスタム変数（より多くのeVar）をより柔軟に使用できます。
   * 一意性を超えました：[!DNL Adobe Analytics] インターフェイスでは、特定の期間に1つのディメンション内で500,000個の一意の値のみを表示できます。 これを超えると、値はインターフェイス内で「一意の超過」または「低トラフィック」としてグループ化されます。 これらの値は、バックエンド（Data Warehouse、データフィード）では引き続き利用できますが、インターフェイス内で視覚化することはできません。 詳細なデータ（ユーザーID、PSNなど）がある場合は、容易にこのレベルに到達できます。 別のレポートスイートを用意しておくと、この問題の解決に役立ちます。

**方法：**&#x200B;新しいAA実装から開始し、単一のグローバルレポートスイートを使用するのは簡単で簡単です。 AA管理UIでグローバルレポートスイート（開発用に1つ、製品用に1つ）を作成し、すべてのプロパティに同じレポートスイート ID （RSID）値を適用するだけです。

プロパティごとに独自のレポートスイートを持つマルチタグ戦略から移行するには、より多くの戦略と計画が必要です。 留意しなければならない考慮事項がいくつかあります。

* 変数の整列（例：プロパティ AのeVar1は、プロパティ BのeVar1と同じデータポイントを取得する必要があります）
* 処理ルール、マーケティングチャネルルール、分類の統合（SAINTとルールビルダー）
* データフィードとデータソースの移行
* カットオーバー日を選択し、すべてのビジネスユーザーにこれを伝える
